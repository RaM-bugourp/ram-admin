## 一、整体架构

```
用户请求 → Nginx → Django (DRF) → MySQL/SQLite
                   ↕ (CORS)
              Vue 3 (Vite) → API 调用
```

### 关键技术栈

| 层级 | 技术选型 | 为什么选它 |
|------|----------|------------|
| 后端框架 | Django 5.2 + DRF | ORM 强大、Admin 后台生态完整 |
| 权限 | RBAC + DataScope | 企业级权限控制的事实标准 |
| 前端 | Vue 3 + Arco Design | 组件丰富、Django 团队友好 |
| 构建 | Vite 6 | 快，HMR 体验好 |
| 状态 | Vuex | 成熟、模块化清晰 |

## 二、后端核心模块

### 2.1 模型基类（BaseAuditModel）

**继承链：**
```
BaseAuditModel
  ↓
  └─ SoftDeleteMixin（软删除）
       └─ is_deleted + deleted_at
  ↓
  ├─ created_at / updated_at（时间戳）
  ├─ created_by / updated_by（用户）
  ├─ owner_organization（组织归属）
  └─ remark（备注）
```

**为什么这样设计？**
- `is_deleted` 做软删除，不真实删除数据，可追溯可恢复
- `created_by / updated_by` 记录责任人，方便审计
- `owner_organization` 用于数据权限隔离

### 2.2 ViewSet Mixin 组合

```python
class ArticleViewSet(
    AuditOwnerPopulateMixin,  # ① 自动填充 created_by / owner_organization
    SoftDeleteMixin,          # ② 查询过滤已删 + perform_destroy 软删除
    ActionSerializerMixin,     # ③ 按 action 选择不同序列化器
    ModelViewSet              # ④ 基础 CRUD
):
    serializer_class = ArticleDetailSerializer  # 兜底
    list_serializer_class = ArticleListSerializer  # list action 用这个
```

**Mixin 执行顺序（在 DRF 源码中）：**
```
请求 → authenticate() → permission.check() → get_queryset() 
    → get_serializer() → perform_create()  ← AuditOwnerPopulateMixin 在这里
    → serializer.save() 
    → perform_update()  ← AuditOwnerPopulateMixin 在这里
```

### 2.3 RBAC 权限体系

**三层模型：**
```
User ←→ UserRole ←→ Role ←→ RolePermission ←→ Permission
                 ↓
          Organization（数据范围）
```

**权限检查时机：**
```
请求进来
  → Middleware（SessionMiddleware 设置 session）
  → Authentication（SessionAuthentication 填充 request.user）
  → Permission（RBACPermission 检查权限）
  → Throttling（限流）
  → View
```

**两种权限模式对比：**
| 模式 | 做法 | 优势 | 劣势 |
|------|------|------|------|
| URL 匹配 | Permission 表存 url_pattern | 接入简单 | 改了 URL 要同步改权限 |
| 权限码 | Permission 表存 code + 装饰器 | 代码即权限，git 追踪 | 需要显式声明 |

**DataScope 数据权限：**
```python
def get_queryset(self):
    queryset = super().get_queryset()
    if user.is_superuser:
        return queryset
    
    for role in user.roles:
        if role.data_scope == 'SELF':
            return queryset.filter(created_by=user)
        elif role.data_scope == 'DEPT':
            return queryset.filter(owner_organization__in=user_orgs)
    return queryset
```

### 2.4 操作日志中间件

**中间件在 Django 中的位置：**
```
request → M1.process_request → M2.process_request → ... → View
         ← M1.process_response ← M2.process_response ← ... ← response
```

**中间件设计的精髓：**
- 不侵入视图代码，所有请求自动记录
- `process_request` 准备数据（存到 request 对象）
- `process_response` 记录日志（请求完成后）
- 异常用 try/except 包裹，不影响主流程

## 三、前端核心模块

### 3.1 Axios 封装

**为什么封装 axios？**
```
直接用：每次都要处理 Token 注入、401 跳转、错误提示
封装后：request(config) 自动完成，API 函数只写业务逻辑
```

**响应拦截流程：**
```
response.data 有 results？
  → 是：分页响应 { count, next, previous, results }，返回整个对象
  → 否：直接返回 data
```

**401 处理特别注意：**
- 登录页本身不发请求 → 不会触发 401 拦截
- `router.currentRoute.path !== '/login'` → 防止死循环

### 3.2 Vuex 状态管理

**为什么用 Vuex 而不是 localStorage？**
- localStorage 需要手动序列化/反序列化
- Vuex 自动响应式，组件用 computed 读取
- Vuex DevTools 可以调试 state 变化
- mutation 有记录，bug 容易追溯

**store 设计原则：**
- user store：登录状态 + 用户信息 + 权限码
- menu store：菜单树（只从后端拉取一次）
- 不在 store 里缓存业务数据（列表数据每次重新拉取）

### 3.3 路由守卫

**beforeEach 是整个权限系统的入口：**
```
用户访问 /dashboard
  ↓
beforeEach 检查 store.state.user.id
  ↓ 没有 → getUserInfo() → 再判断
  ↓ 有 → 检查菜单是否已加载
    ↓ 没加载 → 加载菜单树 + 动态注册路由
    ↓ 已加载 → 放行
```

**动态路由的原理：**
- `router.addRoute()` 可在运行时添加路由
- 菜单树 → 路由数组的转换（递归拼接路径）
- 页面刷新 → state 清空 → 重新拉菜单 → 重新注册

## 四、面试高频问题

### Q1：Django 的 ORM 查询优化手段有哪些？

**A：**
1. `select_related()`：一对一/外键关联查询（JOIN），减少 N+1
2. `prefetch_related()`：多对多/反向查询，预加载避免 N+1
3. `only()` / `defer()`：只查需要的字段，减少数据传输
4. `values()` / `values_list()`：只查特定字段，返回字典/元组
5. `exists()` / `count()`：判断存在性用 exists 比 len() 快
6. 数据库索引：为高频查询字段加索引（`models.Index`）

### Q2：DRF 的认证（Authentication）和权限（Permission）有什么区别？

**A：**
- Authentication 回答：你是谁？（从请求中提取 user）
- Permission 回答：你有没有权限？（基于 user 判断）

执行顺序：先认证拿到 user → 再权限检查

### Q3：Vue 3 相比 Vue 2 有什么改进？

**A：**
1.  Composition API：逻辑复用更灵活（mixin 的问题是有命名冲突、来源不清晰）
2. 性能：Virtual DOM 重写，编译时优化（静态提升、缓存事件处理）
3. TypeScript 原生支持：Vue 3 本身用 TS 写
4. Tree-shaking：按需引入，体积更小
5. 更好的响应式：`ref` / `reactive`，解决了 Vue 2 响应式的坑（如直接改数组索引不触发更新）

### Q4：为什么要用 RESTful API 设计？

**A：**
- 语义清晰：GET 查、POST 增、PUT/PATCH 改、DELETE 删
- 统一接口：任何客户端（Web / App / 小程序）用同一套 API
- 无状态：每个请求自带所有信息，服务端不需要存 session 状态（但本框架用的是 Session 认证）
- 资源化：URL 表示资源 `/api/article/` 而不是 `/api/getArticle`

### Q5：如何防止 SQL 注入？

**A：**
- Django ORM 自动做参数化查询，不用担心 SQL 注入
- 绝对不要用字符串拼接的方式执行 SQL
- 如果必须 raw SQL：用 `cursor.execute(sql, [params])` 参数化
